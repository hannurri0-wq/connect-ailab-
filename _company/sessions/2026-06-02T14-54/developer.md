### 📄 Mini-Deep Dive Report 결제 플로우 통합 테스트 스위트 (E2E Test Suite)

**목표:** PayPal 연동 기반의 $29~$49 결제 프로세스에 대한 End-to-End 안정성 검증 및 리스크 식별.
**가정 환경:** Mocked Payment Gateway API, Stable Network Environment(지정한 실패 시나리오 제외), Database Connection Pool 정상 작동.

#### 🛠️ 테스트 구성 요소 정의 (Prerequisites)

1.  **Test Case Manager:** 모든 테스트 케이스의 실행 순서 및 기대 결과를 관리하는 모듈.
2.  **API Mock Layer:** 실제 PayPal API 대신, 지정된 응답 코드(Success/Failure/Timeout)를 강제할 수 있는 레이어 구축. (테스트 용이성 확보)
3.  **State Tracker:** 세션 시작부터 결제 완료까지의 모든 상태 변화와 데이터 저장 여부를 기록하는 로직.

#### 🔴 Part 2: 실패 주입 시나리오 테스트 (Failure Injection & Resilience)

이 부분이 가장 중요합니다. 실제 운영 환경에서 발생 가능한 모든 예외 상황을 가정하고 테스트해야 합니다.

| ID | 시나리오명 | 유발 원인 (Injection Point) | 예상 결과 (Expected Result) | 시스템의 필수 대응 (Required System Behavior) |
| :--- | :--- | :--- | :--- | :--- |
| **F-001** | **[API] 결제 게이트웨이 실패** | PayPal Mock API가 'Invalid Credentials' 오류 코드 반환. | 🚫 결제 프로세스 즉시 중단, 사용자에게 친절한 에러 메시지 제시. | ① `Error Message Component (EMC)` 표시(현빈님 가이드라인 준수). ② **원인 분석:** "결제 정보 확인이 필요합니다." ③ 재시도 CTA 버튼 활성화. |
| **F-002** | **[Network] 네트워크 단절/타임아웃** | PayPal API 호출 직후, 서버 응답을 받기 전에 연결 끊김 발생 (Timeout). | 🚫 결제 프로세스 중단 및 사용자에게 '네트워크 오류' 고지. | ① 트랜잭션 상태를 `PENDING`으로 임시 저장. ② 재시도 로직 구현(클라이언트/서버 모두에서 재시도 시한 설정). ③ *중복 결제를 막는 멱등성 키* 검증 필요. |
| **F-003** | **[System] DB 기록 오류** | PayPal은 승인 성공했으나, 백엔드 로직이 구매 데이터를 DB에 저장하는 단계에서 에러 발생 (DB Write Failure). | ⚠️ 사용자에게는 "결제는 완료되었으나, 시스템 처리 중입니다. 잠시 후 확인해주세요." 메시지 노출. | ① **가장 중요:** 운영팀/관리자 대시보드에 `MANUAL_REVIEW` 알림을 즉시 트리거해야 함 (실제 매출 누락 방지). ② 사용자에게는 결제 영수증 번호를 제공하여 문의 유도. |
| **F-004** | **[Input] 필수 입력값 누락/유효성 검사 실패** | 사용자가 이메일 주소를 비워두거나, 가격 선택 버튼을 누르지 않음. | 🚫 결제 플로우 진입 전 단계에서 폼(Form) 유효성 검증 오류 메시지 노출. | ① 브라우저 기본 Validator가 아닌, 커스텀 `EMC` 사용. ② 필수 필드에 대한 구체적인 설명 제공 (예: "리포트를 받으려면 정상 메일 주소가 필요합니다."). |
| **F-005** | **[Security] 세션 토큰 만료/변조 시도** | 결제 페이지 진입 후 장시간 방치되어 세션이 만료되거나, 클라이언트 측에서 변조된 요청을 전송. | 🚫 시스템은 강제로 사용자 로그아웃 및 리다이렉트. | ① 모든 API 요청에 대해 `X-Session-Token` 검증 필수. ② 토큰 유효기간(TTL) 명확히 설정하고 만료 시점을 UX적으로 고지해야 함. |

### 💡 통합 테스트 결과 보고서 (Summary Report)

**전반적인 평가:**
기능적 측면에서는 T-001~T-003의 성공 경로가 안정화되었습니다. 하지만, **Part 2의 실패 주입 시나리오에서 심각한 리스크 포인트들이 발견되어 전면적인 코드 보강이 필요합니다.**

**📌 주요 발견된 리스크 (High Priority Fixes):**

1.  **멱등성(Idempotency) 처리 미흡 (Critical):** F-002와 관련된 문제로, 네트워크 단절 후 재시도 시 사용자가 이중 결제를 할 위험이 있습니다. **PayPal API 호출 전, 트랜잭션 고유 ID를 생성하여 요청에 포함하고 서버 레벨에서 이를 강제 검증하는 로직을 최우선으로 구현해야 합니다.**
2.  **비동기 상태 관리 취약 (High):** F-003과 같이 DB 쓰기 오류가 발생했을 때, 사용자에게만 에러를 보여주고 백엔드에 알림이 가지 않는 구조는 치명적입니다. **결제 완료 후 로직은 반드시 '트랜잭션 큐(Message Queue)' 시스템을 거쳐 비동기적으로 처리하고, 실패 시 담당자에게 실시간 알람을 보내는 구조로 변경해야 합니다.**
3.  **오류 메시지 통일성 부족 (Medium):** 현빈님이 제공한 법적 고지사항과 Writer님의 강렬한 카피가 결합되는 지점에서, 시스템 오류 발생 시 사용자 경험(UX)이 일관되지 않을 수 있습니다. 모든 에러 상황은 **'공감-원인 분석-해결책 제시'의 3단계 구조**를 벗어나지 않도록 `EMC` 컴포넌트 로직을 더욱 강화해야 합니다.
