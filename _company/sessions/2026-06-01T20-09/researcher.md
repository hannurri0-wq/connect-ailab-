## 🚨 핵심 장애 시나리오 리서치 보고서 (영상 후크용)

### 🔴 시나리오 1: '돈은 빠졌는데 물건이 오지 않는' 비동기 결제 데이터 불일치
**[주요 약점]**: Saga 패턴의 보상 트랜잭션(Compensation Transaction) 로직 미흡 및 분산 트랜잭션 감지 실패.

| 항목 | 내용 | 활용 가이드 (영상 후크) |
| :--- | :--- | :--- |
| **🎣 후크 제목** | "결제는 성공했는데, 물건 재고가 사라진 개발자가 겪은 악몽" | 가장 직관적이고 돈과 관련된 손실을 다루어 시청자의 이탈 방지. (높은 공포감) |
| **📝 상황 전개** | 사용자가 A 서비스(결제)를 통해 B 상품 구매 요청 -> 결제는 성공적으로 완료되어 금액이 즉시 청구됨 (A 처리 완료). 하지만 C 서비스(재고 관리)와 D 서비스(배송 알림) 간의 비동기 메시지 큐 연동 과정에서 지연 또는 에러가 발생함. 재고 차감이 실패했음에도, 사용자에게는 '성공'이라는 최종 응답이 전달됨. (결제 성공 $\rightarrow$ 재고 없음 / 배송 불가 상태). | **[시각 자료]** 결제 완료 화면과 동시에, 백엔드 로그에 `Inventory Update Failed`라는 빨간 경고문이 깜빡이는 장면을 연출. |
| **⚙️ 기술적 약점 (Pain Point)** | 비동기 시스템에서 데이터 일관성(Eventual Consistency)을 유지하기 위한 보상 메커니즘(Compensation Logic)의 부재. '실패했을 때 롤백하는 방법'이 명확하지 않음. | **[해결책 제시]** Saga 패턴 도입 및 트랜잭션 감시/보정 레이어 설계가 필수임을 강조하며, 유료 리포트 구매로 연결. |

### 🟡 시나리오 2: '작은 기능 장애'가 시스템 전체를 마비시키는 연쇄 폭발
**[주요 약점]**: 서비스 간 결합도(Coupling) 과다 및 회로 차단기 패턴(Circuit Breaker Pattern) 미적용.

| 항목 | 내용 | 활용 가이드 (영상 후크) |
| :--- | :--- | :--- |
| **🎣 후크 제목** | "추천 기능 하나가 쇼핑몰 전체를 먹통으로 만든 순간: 캐스케이딩 장애" | '작은 것'이 '큰 재앙'을 만드는 과정을 드라마틱하게 묘사하여 위기감 조성. (높은 긴장감) |
| **📝 상황 전개** | 사용자 접속량이 폭증함. 이로 인해 평소 중요도가 낮았던 B 서비스(예: AI 기반 상품 추천 엔진)에 갑작스러운 트래픽 부하가 걸림. 이 서비스는 성능 저하를 겪고 느린 응답을 반환하기 시작함. 다른 핵심 서비스인 A(상품 조회/장바구니)가 이느리게 돌아오는 B의 응답을 기다리는 동안, 자체적인 타임아웃 처리 없이 자원을 과도하게 소모하며 함께 지연됨. 결국 전체 시스템이 'Timeout' 오류를 내며 대규모 접속 장애 발생. | **[시각 자료]** 트래픽 그래프가 급상승하는 모습 $\rightarrow$ 특정 서비스의 에러율(Error Rate)이 빨간색으로 솟구치고, 주변 서비스까지 영향을 받아 함께 하락하는 애니메이션 연출. |
| **⚙️ 기술적 약점 (Pain Point)** | 모든 서비스 간에 시간 제한(Timeout)과 실패 임계값(Threshold)을 설정하고, 문제가 발생하면 요청을 아예 차단하여 나머지 시스템이 정상 작동하게 하는 독립성 설계가 부재함. | **[해결책 제시]** 마이크로서비스 아키텍처 환경에서 반드시 Circuit Breaker와 Bulkhead 패턴으로 경계를 지어야 함을 강조하며, 리포트 필요성을 제기. |

### 🔵 시나리오 3: '두 사용자 동시 요청' 처리 실패로 인한 재고 초과 판매
**[주요 약점]**: 데이터베이스 트랜잭션 격리 수준(Isolation Level) 미흡 및 Race Condition 방어 로직 부재.

| 항목 | 내용 | 활용 가이드 (영상 후크) |
| :--- | :--- | :--- |
| **🎣 후크 제목** | "마지막 남은 굿즈! 두 명이 동시에 구매했을 때 벌어진 데이터베이스의 거짓말" | '경쟁'과 '극적인 실패'라는 요소를 결합하여 몰입도를 극대화. (높은 흥미와 공포감) |
| **📝 상황 전개** | 재고가 딱 1개 남은 한정판 상품이 있음. A 사용자(브라우저 탭 1)와 B 사용자(모바일 앱)가 거의 동시에 '구매' 버튼을 누름. 두 요청 모두 DB에 접근하여 현재 재고를 확인하는 과정에서, 아직 트랜잭션 커밋되지 않은 상태임. (두 요청 모두 "재고: 1개"로 판단). 결국 두 개의 주문이 성공적으로 처리되지만, 실제 물리적 재고는 0개가 되어 초과 판매(Overselling)가 발생하고, 재고 관리 시스템 전체에 데이터 불일치가 생김. | **[시각 자료]** '재고: 1'을 보여주는 화면이 동시에 두 번의 트랜잭션으로 인해 '구매 성공' 상태로 바뀌는 모습. 백엔드 코드가 병렬적으로 충돌하는 듯한 시각 효과 사용. |
| **⚙️ 기술적 약점 (Pain Point)** | 데이터베이스 레벨에서 Optimistic Locking 또는 Pessimistic Locking을 사용하여 Read-Modify-Write 과정의 원자성(Atomicity)을 보장하지 못했음. 동시성 제어 로직이 가장 취약한 지점임을 인식해야 함. | **[해결책 제시]** 트랜잭션 격리 수준과 락킹 메커니즘에 대한 깊은 이해가 필요하며, 이 분야의 전문 진단 리포트가 유효함을 간접적으로 시사. |
