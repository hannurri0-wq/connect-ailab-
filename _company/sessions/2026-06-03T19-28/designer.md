### 📂 Mini-Blueprint Pro: Danger Red 시뮬레이션 인터랙티브 데크 최종 명세 (TechSpec v2.0)

**목표:** 청중에게 '미조치로 인한 재무적 공포'를 체험시키고, 제품의 필요성을 논리적/감성적으로 각인시킨다.
**형식:** 슬라이드 기반 인터랙티브 프로토타입 (Figma 또는 WebGL 기반)

#### 2. 핵심 컴포넌트 디자인 명세 (Component Specifications)

**A. Danger Red 경고 상태 UI (Critical Component)**
*   **발동 조건:** `calculate_loss_score` 결과가 임계점(Threshold)을 초과했을 때.
*   **시각 효과:**
    *   배경: 짙은 검정색 또는 어두운 회색으로 설정.
    *   컬러 오버레이: `#C8001A` (Danger Red). 이 색상은 배경 전체에 미세한 노이즈/글리치 필터로 깔려 있어야 함.
    *   폰트 변화: 모든 텍스트는 Impact 계열의 볼드체 사용을 기본으로 하며, 크기 변화가 동적이어야 함.
    *   애니메이션 스펙: 경고 메시지 출현 시 **100ms 간격의 깜빡임(Blink)**과 함께 화면이 좌우로 미세하게 진동하는 효과(Shake Effect)를 적용해야 합니다.

**B. 실시간 손실액 표시기 (Live Loss Simulator Display)**
*   **위치:** 데크 화면 중앙 상단, 가장 눈에 띄는 위치.
*   **데이터 바인딩:** API 호출 결과(`Loss Amount`)와 직접 연동되어야 함.
*   **상태 변화 스펙:**
    *   일반 상태: 투명한 배경 위에서 숫자가 서서히 올라가는 카운트업 애니메이션 (Smooth Transition).
    *   위험 상태: 숫자가 빨간색으로 변하며, '!!!!' 경고 마크와 함께 **진동 및 글리치 효과**를 동반해야 함.

#### 3. 개발팀 전달 사항 (Developer Handoff Checklist)

1.  **API Mocking:** 데크 제작 시, 실제 API 호출을 기다리지 않도록 `calculate_loss_score` 함수의 결과를 모킹(Mocking)하여 다양한 스코어 값(0.2 $\rightarrow$ 0.6 $\rightarrow$ 0.9)이 들어왔을 때의 프론트엔드 반응만 완벽하게 구현할 수 있도록 요청합니다.
2.  **사운드 디자인:** 경고음/앰비언스는 반드시 개발 단계에서 별도로 녹음을 진행하고, 특정 상태(Danger Red 진입, 위험도 최대 도달)에 맞춰 트리거되도록 사운드 스펙을 정의해야 합니다.
