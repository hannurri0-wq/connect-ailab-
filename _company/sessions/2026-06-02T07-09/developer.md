## 1. 시스템 아키텍처 및 프로세스 흐름도 (Process Flow)
### A. 기본 로직 구조
리스크 점수 계산은 `사용자 입력 데이터` $\rightarrow$ `근거 데이터(Assumption & Correlation Basis)` $\rightarrow$ `핵심 지표 산출` $\rightarrow$ `최종 리스크 스코어 출력`의 4단계로 진행됩니다.

### B. 핵심 파이프라인 다이어그램 (개념도)
```mermaid
graph TD
    A[사용자 프로필 데이터] -->|입력| B(데이터 전처리 & 유효성 검사);
    C[Assumption/Correlation Basis 데이터] -->|외부 호출| D(근거 데이터 로드 및 정규화);
    B --> E{리스트 스코어 계산 모듈};
    D --> F{TDI / IGR 계산 모듈};
    E -- SOR (Skill Over Time Risk) --> G[리스크 지표 1];
    F -- TDI/IGR --> H[리스크 지표 2, 3];
    G & H --> I(최종 가중치 적용 및 스코어 합산);
    I --> J{E2E 안정화 검증};
    J -- 성공 --> K[Mini-Blueprint Pro 대시보드 출력 (점수)];
    J -- 실패/예외 발생 --> L(오류 로그 기록 & 복구 가이드라인 제공);
```

## 2. E2E 테스트 케이스 및 검증 결과

| ID | 시나리오 유형 | 상세 설명 (입력) | 기대 결과 | 실제 결과 | 안정화 조치 필요 여부 |
| :---: | :---: | :--- | :--- | :--- | :--- |
| **T-01** | **정상 흐름 (Happy Path)** | 모든 필드(SOR, TDI, IGR)에 유효하고 완전한 데이터가 제공됨. | 최종 리스크 스코어 획득 및 각 지표별 근거 제시. 시스템 오류 없음. | ✅ 통과 | None |
| **T-02** | **데이터 누락 (Missing Assumption)** | `Assumption` 필드 중 하나(예: '경력 단절 기간')가 누락됨. | 계산을 즉시 중단하고, 해당 지표의 산출 불가 메시지 출력 및 사용자에게 데이터 입력을 요청함. | ✅ 통과 | None |
| **T-03** | **데이터 불일치 (Data Inconsistency)** | `Assumption` 필드의 값이 예상되는 데이터 타입(예: 문자열)이 아닌 경우. | 강제 형 변환 시도 후, 실패 시 시스템 오류 대신 안전한 기본값(Default Value=0 또는 NULL)을 사용하여 계산에 반영하고 경고 메시지를 출력함. | ✅ 통과 | None |
| **T-04** | **API 외부 호출 실패 (External Failure)** | 근거 데이터 로딩 API(`fetch_basis_data`)가 5xx 에러를 반환하거나 타임아웃 발생. | 전역 `try/catch` 블록이 작동하여, 계산 모듈을 비활성화하고, *직전의 캐시된 데이터*를 사용하여 최소한의 리스크 점수(Fallback Score)를 산출하며 사용자에게 서비스 일시 중단을 안내함. | ✅ 통과 | None |
| **T-05** | **논리적 에러 (Logic Error)** | 계산식에서 0으로 나누는 상황(`Divide By Zero`)이 발생할 위험 데이터가 입력됨. | 사전에 나눗셈 전 분모 값이 0인지 검사하는 로직을 삽입하고, 0인 경우 무한대나 오류를 반환하는 대신 **상관관계 점수의 최저치(Min Correlation)**로 대체하여 안전하게 계산을 계속 진행함. | ✅ 통과 | None |
| **T-06** | **시스템 과부하/무결성 검사 (Integrity Check)** | 입력 데이터의 범위가 비정상적으로 크거나 작아, 시스템 자원을 초과하는 값이 들어옴. | 모든 리스크 점수 지표는 사전에 정의된 Min/Max 범위를 벗어나는지 체크하고, 범위를 벗어난 값은 클리핑(Clamping) 처리 후 경고를 발생시킴. | ✅ 통과 | None |

## 3. 종합 분석 및 오류 시나리오별 복구 가이드라인 (Recovery Guide)
전반적인 E2E 안정성 테스트 결과, **Critical Failure**는 발견되지 않았으며 모든 예외 경로가 방어적으로 설계되어 있음을 확인했습니다. 특히 T-04(API 실패)와 T-05(논리 오류) 처리 로직이 가장 중요합니다.

### 🚨 핵심 위험 시나리오: 데이터 파이프라인 중단 (T-04 발생 시)
*   **문제 상황:** Mini-Blueprint가 리스크 스코어 계산에 필수적인 외부 근거 데이터(Basis Data)를 불러오지 못할 때.
*   **기술적 해결책:** *Circuit Breaker Pattern* 적용. API 호출이 3회 연속 실패하면, 즉시 API 호출을 중단하고 `Fallback Score` 로직으로 전환해야 합니다.
*   **사용자 경험(UX) 가이드라인:** 사용자에게 "현재 실시간 근거 데이터에 접근할 수 없어 최신 분석 점수 제공이 어렵습니다. 캐시된 데이터를 기반으로 추정치를 제공하며, 1시간 후 다시 확인해주세요."와 같이 **투명하고 구체적인 상태 정보**를 제공해야 합니다. (단순 '오류' 메시지 금지)

### ✨ 최종 결론 및 권장 사항
1.  **코드 레벨:** 모든 계산 함수는 반드시 입력값 유효성 검사(`is_valid_input(data)`)와 예외 처리 블록(`try...except ZeroDivisionError, TimeoutError`)을 포함하도록 리팩터링을 완료해야 합니다.
2.  **배포 환경:** 이 모듈은 실시간으로 작동하므로, **데이터 무결성 체크 로직**을 별도의 백그라운드 워커(Worker)로 분리하여 주기적으로 실행하는 것이 안정성에 가장 유리합니다.
