## 🎣 프로젝트 계획서: 낚시앱 (Fishing App)
### 목표 마일스톤 정의: Core Data & User Journey 완성
*(다음 주까지 달성해야 할 최소 기능 단위)*

#### I. 기술 스택 플로우 및 아키텍처 검증 (Tech Stack Flow)
**Goal:** 시스템의 안정성을 최우선으로 확보하고, 외부 API 연동에 의한 데이터 파이프라인을 구축하는 데 초점을 맞춥니다.

| 단계 | 구성 요소 (Component) | 사용 목적/역할 | 기술적 요구사항 및 검증 항목 |
| :--- | :--- | :--- | :--- |
| **1. 클라이언트 레이어** | Mobile Client (Expo RN/React Native) | 사용자 인터페이스 제공, API 호출 요청 전송. | 📍 GPS 권한 관리 로직 구현. 오프라인 상태에서의 데이터 캐싱 및 복구 메커니즘 필수. |
| **2. API 게이트웨이** | Backend Gateway (Node.js/Express 또는 Python/Flask) | 클라이언트의 모든 요청을 수신하고, 내부 서비스 레이어로 분배. 인증(AuthN/AuthZ) 처리를 전담함. | 🔒 JWT 기반 토큰 검증 필수. Rate Limiting 적용 및 입력값 유효성 검사(Validation). |
| **3. 비즈니스 로직** | Service Layer (Core Business Logic Module) | 실제 데이터 처리, 계산(예: 어획 확률 점수 산출), 외부 API 호출을 조정함. | ⚙️ **핵심:** `FishingLogicService` 구현. 지역별/계절별 데이터를 조합하여 '어획 가능성 스코어'를 산정하는 알고리즘 정의 및 단위 테스트 작성. |
| **4. 데이터 저장소** | Database (PostgreSQL 또는 MongoDB) | 사용자 정보, 활동 로그(Catch Log), 지역별 기본 메타데이터를 영구적으로 저장함. | 💾 Schema 설계 검증 완료. `user_logs` 테이블에 시간 순서 및 위치 기반 인덱싱 필수. |
| **5. 외부 연결** | External API Connector (Weather/Current) | 날씨, 수온, 조류(Tide) 등 실시간 데이터를 가져와 서비스 로직에 투입함. | ⚠️ 실패 시 폴백(Fallback) 로직 구현: 외부 데이터가 없어도 앱이 크래시하지 않도록 기본값을 설정해야 함. |

#### II. 최우선 핵심 기능 3가지 (Top 3 Priority Features)
**Focus:** 이 세 가지는 '사용자 액션'과 '데이터 생성/활용'의 사이클을 완성하는 데 필수적입니다.

1.  **[핵심] 실시간 환경 데이터 연동 및 어획 가능성 스코어 산출 (API + Logic)**
    *   **기술 검증:** 외부 API (날씨, 조류) → Service Layer에서 데이터를 받아와 🎣 `FishingLogicService`를 통해 점수화(Score)합니다.
    *   **기능 목표:** 단순히 날씨 정보를 보여주는 것을 넘어, "현재 위치/시간대 기준 어획 가능성: **B+ (75%)**"와 같이 수치로 사용자에게 공포감과 기대를 동시에 전달해야 합니다.
2.  **사용자 활동 로깅 및 지도 기반 기록 시스템 (Database + Client)**
    *   **기술 검증:** 사용자가 낚시를 마친 후, `(위도, 경도, 시간, 어획 품목, 수량)`을 백엔드에 전송하고 DB에 저장합니다.
    *   **기능 목표:** 지도 뷰에서 과거의 활동 기록(Catch Log)이 포인트로 찍히고, 이 로그가 나중에 **"당신이 자주 가는 곳"**이라는 개인화된 인사이트를 제공해야 합니다. (데이터 자산화).
3.  **지역별/어종별 '미발견' 정보 및 힌트 제공 (Content + UI)**
    *   **기술 검증:** 데이터베이스에 저장된 공통 메타데이터와, 시간이 지나면서 사용자가 쌓아가는 로그 데이터를 비교하여 "이 지역에서는 이 어종을 아직 많이 발견하지 않았습니다"와 같은 형태의 콘텐츠를 제시합니다.
    *   **기능 목표:** 단순한 지식 제공(Education)을 넘어, **'내가 놓치고 있는 기회(Missed Opportunity)'**라는 심리적 자극에 초점을 맞춰야 합니다.

#### III. Mini-Blueprint Pro 관련 작업 보류 가능성 검토
✅ **결론: 코어 로직 구현은 분리하여 진행 가능합니다.**

*   **기술 관점:** MBP의 LPO 컴포넌트 개발과 낚시앱의 `FishingLogicService`는 완전히 다른 기술 영역(마케팅 퍼널 vs. 지리 정보/데이터 처리)입니다. 두 프로젝트를 별도의 마이크로 서비스 아키텍처(MSA) 개념으로 분리하면, **MBP 관련 코드 구현을 잠정적으로 보류해도 낚시앱의 개발 속도에 치명적인 영향을 주지 않습니다.**
*   **전략적 제언:** MBP는 현재 *최종 검증 및 마케팅 콘텐츠 고도화* 단계(영숙 보고서 참조)로 전환하여, **'완성된 기능 테스트 환경'을 유지하는 데만 리소스를 집중하고**, 실제 코드 개발 작업은 다음 주부터 재개하는 것이 가장 효율적입니다.
