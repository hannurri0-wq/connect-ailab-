### 📝 Mini-Deep Dive Report (MDR) E2E 통합 테스트 시나리오 (v1.0 Draft)

**목표:** 결제 플로우의 모든 성공 및 실패 경로를 커버하고, 시스템 안정성(Resilience)을 확보하여 '미조치 손실 공포'를 구매 행동으로 이어지게 한다.
**테스트 범위:** Frontend $\to$ Backend API ($\text{/api/v1/purchase}$) $\to$ Payment Gateway (PG) $\to$ Report Generation Service $\to$ Database

| **TC-03** | ❌ 시스템 Failure: 네트워크 타임아웃 | 1. 결제 요청 $\to$ 2. 백엔드 서버 또는 PG사 API 호출 과정에서 강제 연결 끊김(Timeout). | 클라이언트에게 명확한 '네트워크 오류' 메시지 표시 및 **재시도 버튼** 제공. (시스템이 트랜잭션을 임의로 취소하거나 승인하지 않아야 함). | **[필수]** 재시도 시, 이전에 발생한 미완료 상태를 감지하고 사용자에게 경고하며 다시 시작하게 유도해야 함. |
| **TC-04** | 💣 백엔드 Failure: 보고서 생성 실패 (Critical) | 1. 결제 승인 완료 $\to$ 2. Backend가 PDF/보고서 파일을 생성하는 과정에서 내부 오류 발생 (e.g., 라이브러리 충돌). | 트랜잭션은 일단 성공(PG사 승인 상태 유지), 하지만 사용자에게 **'시스템 점검 중, 잠시 후 보고서를 받으세요.'**와 같은 명확한 안내 메시지 제공 및 백엔드에서 비동기적으로 실패 알림 트리거. | **[최우선]** 결제 금액은 환불(Refund) 처리되어야 하며, 오류 원인을 추적할 수 있는 로그가 남아야 함. (멱등성 체크 필수). |

| **TC-06** | 🔒 Security Test: 비정상 접근/속도 제한 | 짧은 시간 동안 동일 IP 또는 사용자 계정에서 과도하게 많은 결제 시도를 반복한다 (DDoS 방지 목적). | 시스템은 일정 임계치 이상의 요청을 감지하고, 일시적으로 API 접근을 차단(Rate Limiting)하며 사용자에게 남은 시도 횟수를 고지. | **[필수]** 단순한 에러 코드를 넘어서, '잠시 후 다시 시도해 주세요'와 같은 부드러운 UX 안내가 필요함. |
| **TC-07** | 🧪 Edge Case: 쿠폰/할인 적용 오류 | 존재하지 않거나 만료된 쿠폰 코드(e.g., `INVALID_CODE`)를 결제 직전에 입력한다. | 시스템은 실시간으로 해당 코드를 검증하고, 사용자가 인지하기 쉬운 방식으로 '유효하지 않은 할인 코드입니다' 메시지를 표시해야 함. | **[필수]** 가격 계산 로직이 정확히 작동하는지(최종 결제 금액 재계산) 반드시 확인해야 합니다. |

| **TC-09** | 🗑️ Rollback Test: 강제 취소/환불 요청 | PG사 승인 직후, 사용자가 이탈하여 '취소' 버튼을 눌러 환불을 요청한다 (시스템 또는 사용자 측). | 백엔드는 트랜잭션을 성공적으로 'VOID' 처리하고, 데이터베이스의 상태를 정확하게 업데이트해야 함. (미사용 리소스 정리). | **[필수]** 취소된 트랜잭션 기록이 남아야 하며, 해당 금액을 환불하는 PG API 호출이 이루어져야 함. |
| **TC-10** | 🔗 Flow Integration: 구매 후 데이터 연동 테스트 | Mini-Deep Dive Report를 성공적으로 받은 사용자에게 다음 Funnel 단계(예: 유료 코스 안내)로 자연스럽게 연결되는지 확인한다. | 보고서 다운로드 페이지/대시보드에 '다음 단계 추천 콘텐츠'가 노출되고, 해당 CTA가 다른 비즈니스 플로우와 연동되어야 함. | **[필수]** 단일 트랜잭션으로 끝나는 것이 아니라, 다음 구매를 유도하는 *흐름* 자체가 테스트 대상입니다. |

## 🎯 목표
Mini-Deep Dive Report 구매 플로우의 안정성 및 사용자 경험(UX) 극대화. 모든 예외 상황을 포괄하여 시스템 아키텍처를 검증한다.

## 🧪 핵심 원칙
1. **원자성 (Atomicity):** 결제 성공과 보고서 생성/사용자 데이터 저장이 하나의 트랜잭션 단위로 묶여야 한다. 하나라도 실패하면 전부 롤백되어야 한다.
2. **멱등성 (Idempotency):** 동일한 요청(Request)을 여러 번 보내더라도, 시스템 상태는 한 번의 성공으로 유지되어야 한다 (이중 청구 방지).
